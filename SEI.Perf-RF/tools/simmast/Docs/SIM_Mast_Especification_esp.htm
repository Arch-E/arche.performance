Simulator of MAST Models

Sim_Mast es una herramienta que ejecuta la simulación de un modelo MAST, proporcionando información sobre:

 

-         El cumplimiento de los requerimientos temporales establecidos en el modelo.

-         La respuesta temporal que proporciona en el peor, mejor o caso promedio.

-         Los niveles de utilización de los recursos.

-         Trazas de ejecución.

 

Objetivos

 

Los objetivos para los que se ha desarrollado el Simulador son:

 

-         Estimar el comportamiento del sistema modelado en aquellos casos en que no existen herramientas analíticas de análisis.

-         Evaluar la eficiencia de otros métodos de diseño y análisis disponibles.

-         Validar estrategias de análisis que se estén desarrollando, identificando, en su caso, las situaciones de fallo que presenten.

 

Casos de uso

 


La herramienta se destina a dos posibles usuarios:

-         El diseñador que necesite conocer el comportamiento de un sistema cuyo comportamiento temporal ha sido previamente modelado con Mast. El diseñador necesitará la ejecución de la simulación del modelo y la posterior visualización de los resultados que se obtengan.

 

-         Herramientas automáticas destinadas a implementar procedimientos de diseño o de evaluación de otras herramientas de análisis. En este caso, se invocará iterativamente la ejecución de la simulación sobre conjuntos de modelos que genera la herramienta.  

 

 

Componentes

 

La herramienta se proporciona como un conjunto de cuatro programas ejecutables:

 

-         Sim_Mast_Launcher: Es una sencilla aplicación GUI que permite invocar la macro que define los parámetros de ejecución de una simulación.

 

-         Sim_Mast_Simulator: Es el programa que ejecuta la simulación del modelo.

 

-         Mast_Results_Displayer: Herramienta GUI que visualiza gráfica y tabularmente los resultados de la simulación.

-         Mast_Traces_Result: Herramienta GUI que visualiza gráficamente las trazas generadas por la simulación.

 


 


Interfaz con usuario

 

El módulo Sim_Mast_Launcher es un programa GUI que permite a través de una ventana seleccionar el modelo MAST que va ser estudiado, seleccionar los parámetros de simulación  e invocar sobre él las diferentes herramientas.

 

La interfaz de usuario se basa en una ventana gráfica desde la que se puede invocar diferentes herramientas:

 

Main_Windows: Contiene controles para definir todos los elementos que son necesarios para lanzar una ejecución de simulación.

 

 Directory: Campo de texto que define el directorio que se utiliza para construir el pathname de los ficheros Model File, y los derivados de Campo Results File.

 

Model File: Campo de texto que define el fichero del modelo MAST que se utiliza como entrada al simulador. El pathname absoluto del Fichero de entrada es <Directory>&’\’&<Model File>&”.mdl”. Por ejemplo en el caso de la figura sería: “c:\Temp\Telerobot.mdl”

 

Results File: Campo de texto que define los nombres de los ficheros que se generan como salida de la ejecución de Simulador. El pathname absoluto de los ficheros de salida serían <Directory>&’/’&<Result Files>&{”.res”|”log”|”bak”}. Por ejemplo en el caso de la figura, el fichero de resultados sería: “c:\Temp\Telerobot_1.res”

 

Time: Campo de texto que define el tiempo de simulación máximo en el que se concluye la ejecución de la simulación. En el caso de algunos profiles es directamente el tiempo de simulación, en otros casos es un tiempo de time out para el caso de que no se haya satisfecho el criterio ordinario de finalización.

 

Profile:  Etiqueta que especifica el perfil de resultados que se utiliza en la simulación. Puede tomar los valores SCHEDULABILITY, VERIFICATION, EXHAUSTIVE.

 

 

Buffer: Define la longitud en bytes del buffer de logger que se utiliza. En el caso de buffer que se gestiones de forma acotado o circular, es la longitud del buffer, mientras que si se gestiona como ilimitado, representa la longitud de los bloques con que se almacenan en disco. Puede tomar los valores enumerados 512, 2K, 8K y 32K

 

New model: Botón que lanza el navegador que ayuda a la selección del fichero que se utiliza como modelo. 

 

Default: Botón que establece como nombre del fichero de resultados el nombre del fichero de modelo seguido de un guión bajo y de un número,esto es, <Model File>&’_’&<n>’. La finalidad de este criterio es que la generación de resultados no elimine resultados previos.

 

From: Botón que establece como nombre de modelo el fichero de respaldo que ha sido origen del fichero de resultados establecido. Este fichero es el que tiene como nombre el mismo que el fichero de resultados con la extensión ‘.bak’.

El botón se deshabilita cuando se establece un nombre de fichero de resultados y no existe el correspondiente fichero de respaldo. Se habilita cuando se ejecuta la simulación y se crea el fichero de respaldo, o cuando se cambia el nombre de ficheros de resultados y existe el fichero de respaldo.

 

Edit model: Botón que lanza el editor del fichero de modelo (programa en desarrollo, por ahora establecerlo como inhabilitado). El editor lee el fichero establecido como modelo sea cual sea su extensión {‘.mdl’|’.bak’} y siempre guarda el fichero editado con extensión ‘.mdl’ 

 

Execute: Botón que ejecuta la simulación del fichero establecido como modelo.

Solo está habilitado si la extensión del fichero modelo es ‘.mdl’

 

View Results: Botón que lanza el visor de resultados.

Solo está habilitado si existe el fichero de resultados con la extensión ‘.res’.

 

Display Traces: Botón que lanza el visualizador gráfico de trazas.

Solo está habilitado si existe el fichero de resultados con la extensión ‘.log’.

 

Exit: Botón que finaliza el programa. 

 

Help: Botón que visualiza una página .htlm con ayuda para manejar este programa.

 

 

 

Profiles

 

Establece el objetivo para el que se invoca la simulación del modelo. En función del profile se definen las magnitudes estadísticas que se evaluan, los mensajes de logging que se incluyen, la forma de gestionar el buffer de mensajes y el criterio de finalización de la simulación. 

 

SCHEDULABILITY_PROFILE: Tiene el objetivo de generar una información comparable a la que genera la herramienta de análisis de planificabilidad:

-   Monitoring:

o       Los eventos que tienen asignados Timing Requirement y los que son salida de segmento.

o       Los eventos que representan fin de una transacción (van incluidos en los que son salida de segmento).

o       El % de utilización de los recursos.

-       Logging:

o       Se generan mensajes cuando se incumplen un Timing Requirement.

-       Buffer Managing:

o       BOUNDED mode.

-       Finalizing criterion:

o       Cuando  cada cada transacción se haya concluido por cada uno de sus modos al menos un numero mínimo de veces, o como time out, cuando se alcance un tiempo máximo de simulación.

 

VERIFICATION_PROFILE: Tiene el objetivo de localizar y justificar las situaciones en las que fallan el cumplimiento de los requerimientos temporales.

-            Monitoring:

o     Los eventos que tienen asignados Timing_Requirement y los que son salida de segmento.

o     Monitoriza el % de utilización de los recursos

-         Logging:

o     Genera un mensaje si se incumple un Timing_Requirement.

o     Genera un mensaje por cada Flow_Event que se genera.

-         Buffer Managing:

o      CIRCULAR mode.

-         Finalizing criterion:

o       Cuando  cada cada transacción se haya concluido por cada uno de sus modos al menos un numero mínimo de veces, o cuando se incumple un Timing Requirement, o como time out, cuando se alcance un tiempo máximo de simulación.

 

 

EXHAUSTIVE_PROFILE: Tienen el objetivo de generar una información exhaustiva  sobre la ejecución de la simulación.

-            Monitoring:

o     Los eventos que tienen asignados Timing_Requirement y los que son salida de segmento.

o     Monitoriza el % de utilización de los recursos

-         Logging:

o     Genera un mensaje si se incumple un Timing_Requirement.

o     Genera un mensaje por cada Evento que se genera.

o     Genera un mensaje por cada cambio de estado de todos los Processing Resources, Shared_Resources ,Scheduling_Server, timers, activities.

-         Buffer Managing:

o      UNBOUNDED mode.

-         Finalizing criterion:

o  Cuando  cada cada transacción se haya concluido por cada uno de sus modos al menos un numero mínimo de veces, o cuando se incumple un Timing Requirement, o como time out, cuando se alcance un tiempo máximo de simulación.

 

Restrictions

 

-          Modelos no implementados

o       Network con tipo de transimisión full-duplex

o       Drivers de tipo Character_Packet_Driver.

 

-          Restricciones a los modelos

o       Todos los canales de salida de un Query_server tienen que entrar en actividades.

o       Cada Event_Handler tipo Simplex_FP_Packet_Dispatcher (modelo de la actividad que realiza el envio de un mensaje en una network) tiene que tener un server independiente.

o       La operación que lleva a cabo un Simplex_FP_Packet_Dispatcher ha de ser simple.

o       Un Ev_H tipo Simplex_FP_Packet_Dispatcher ha de ir siempre entre dos Ev_H tipo Activity.

o       Un Scheduling Server de tipo Sporadic ha de tener un número de Max_Pending_Requirements mayor que 1.
